METIS.AI 아키텍처 문서

멀티테넌트 AgentOps 거버넌스 SaaS — Turborepo 기반 NestJS + Next.js + BullMQ 아키텍처.

1. 개요

METIS.AI는 AI Agent의 실행을 평가(4-Gate)·통제(정책)·배포(릴리스)하는 거버넌스 플랫폼입니다. 단일 코드베이스(Turborepo)에 3개 런타임(web·api·worker)과 3개 공유 패키지(database·types·config/ui)로 구성됩니다.

아키텍처 원칙
  • 도메인 모듈화: NestJS 모듈 단위로 거버넌스/평가/실행/릴리스 도메인 분리
  • 정책 구동(config-driven): 평가 Gate 기준을 코드가 아닌 DB 정책으로 통제
  • 테넌트 격리: 모든 데이터 접근을 tenantId로 스코핑
  • 비동기 분리: 무거운 작업(import·실행·replay·canary)은 BullMQ 큐로
  • 안전한 fail: 평가/감사 실패가 핵심 흐름을 막지 않도록 격리(try/catch)

7. 평가 엔진 (4-Gate) 핵심

모든 Agent 응답은 EvaluatorService.evaluate()를 통과합니다. 4개 Gate를 독립 채점한 뒤 정책 가중치로 종합 점수(0–100)를 산출합니다.

품질 Gate정확도·환각·응답품질
+ LLM Judge
보안 Gate입력위협·출력유출·툴체인
이상탐지 GateZ-Score·IQR·추세
비용 Gate효율·지연·토큰
▼ 정책 가중치 적용 (기본 40/30/15/15, 자동 정규화)
종합 점수 + 하드게이트품질 F → ≤40 · 보안 critical → ≤cap · 등급 A~F

설계 포인트

  • 정책 구동: 가중치/임계값을 EvaluationPolicyService가 DB에서 로드(5분 캐시). 미설정 시 기본값=기존 동작.
  • 하드게이트: 가중평균을 상한으로 덮어쓰는 캡 — 사실오류·보안유출은 다른 점수와 무관하게 F 보장.
  • LLM Judge: factual_accuracy 50% 가중 + 보정 앵커 프롬프트 + temperature 0(일관성) + 예산캡 + Anthropic→OpenAI→Layer0 폴백.
  • 적응형 샘플링: 건강할 때 Judge 호출 빈도를 floor(10%)까지 자동 절감.

9. 릴리스 엔지니어링

단계 메커니즘 안전장치
Replay 과거 입출력을 후보 버전에 재생 회귀/개선 verdict
Shadow 운영 트래픽 그림자 실행(부작용 없음) control vs candidate diff
Canary 트래픽 점진 라우팅(5%→100%) Gate 평가 + 자동 롤백
Promotion 버전 승격/롤백 이력·근거 기록

이 작업들은 Worker의 replay·shadow·canary 큐에서 비동기 처리됩니다.

10. 비동기 처리 (Worker)

API가 BullMQ 큐에 작업을 넣고 Worker가 소비합니다. 큐: pack-import · execution · replay · shadow · canary · auto-actions.

  • 각 작업은 correlationId로 추적되고 완료/실패 시 AuditLog 기록
  • graceful shutdown(SIGINT/SIGTERM) + unhandledRejection/uncaughtException 핸들러
  • 동시성: WORKER_CONCURRENCY(기본 5), canary는 2로 제한

11. 보안 · 멀티테넌시

  • 인증: /auth/login → JWT. HttpOnly 쿠키 + 응답 body Bearer.
  • RBAC: @Roles() 데코레이터 + RolesGuard, 6역할.
  • CSRF: double-submit 쿠키. Bearer JWT 요청은 면제(헤더 인증은 CSRF 벡터 아님).
  • 테넌트 격리: withTenantIsolation(prisma, ctx)로 모든 쿼리를 tenantId 스코핑. tenant 미존재 시 첫 테넌트로 폴백(resolveTenantId).
  • 입력 검증: 전역 ValidationPipe(whitelist + forbidNonWhitelisted) — DTO 밖 필드는 400.
  • 보안 헤더: helmet. 비밀/PII는 보안 Gate가 출력에서 탐지.

12. 핵심 시퀀스

A. Agent 실행 → 평가

Client → API /executions
  → 정책 평가(PolicyService): FAIL이면 차단
  → ExecutionSession(QUEUED) 생성 → BullMQ execution 큐
Worker → runExecution → 스텝 실행 → ExecutionStep 기록
  → EvaluatorService.evaluate(): 4-Gate + LLM Judge(적응형 샘플링)
  → AgentEvaluation 저장 + AuditLog

B. 평가 Gate 설정 변경

Admin → PUT /governance/evaluation-policy (id 제외 페이로드)
  → EvaluationPolicyService.updatePolicy → DB upsert → 캐시 무효화
  → 다음 evaluate() 호출부터 새 가중치 적용(≤5분)

C. 정책 피드백

Admin → POST /governance/policy-suggestions/analyze
  → 30일 AgentEvaluation 로드 → analyzeEvaluations(순수) → PolicySuggestion(PENDING)
Admin → POST .../:id/approve → proposedChanges를 EvaluationPolicy에 적용(APPLIED)

13. 설계 결정 · 트레이드오프

결정 이유 / 트레이드오프
순수 로직 분리(pattern-analysis, adaptive-sampling, computeOverallScore) NestJS/Prisma 없이 단위 테스트 가능. I/O와 분리되어 회귀 검증 용이.
정책 미설정 시 기본값 = 기존 동작 하위호환 보장. 단, 정책 도입 전후 점수가 동일해야 함(테스트로 검증).
적응형 샘플링 인메모리 상태 단순·빠름. 트레이드오프: 서버 재시작 시 100%로 초기화(영속 필요 시 Redis 백업).
평가/감사 실패를 try/catch로 격리 핵심 실행이 막히지 않음. 트레이드오프: silent-fail 위험 → 잘못된 필드 누락이 늦게 발견될 수 있음(코드 감사로 보완).
(prisma as any) 캐스트 생성 타입 지연 대응. 트레이드오프: 컴파일 타임 타입 안전성 일부 포기.
db push(개발) vs migrate(운영) 개발 편의. 운영 배포 시 정식 마이그레이션 이력 권장.
