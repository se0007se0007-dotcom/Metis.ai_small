pnpm 11.7.0 업그레이드 + 라이브러리 버전 고정(metis.ai 빌드 성공本과 동일) 반영 후, 담당자가 사내망에서 수행해야 할 단계별 절차서

대상 CRM사업팀 빌드/배포 담당 리포지토리 metis-ai (Turborepo · pnpm 모노레포) 작성일 2026-06-22 버전 v1.0

0. 한눈에 보기 — 잔여 단계 체크리스트

이번 변경으로 코드/패키지 버전은 모두 반영 완료되었습니다. 사내망에서는 아래 순서대로만 진행하면 됩니다. 1번(락파일 재생성)이 가장 중요하며, 이전 “pnpm build 이슈”의 직접 원인입니다.

  • STEP 1. 사내 Nexus 허용 목록(allow-list)에 pnpm@11.7.0 + 신규 고정 버전 등록 확인
  • STEP 2. pnpm-lock.yaml 재생성 (Nexus 연결 환경에서 pnpm install) 후 커밋 — 필수
  • STEP 3. preflight(폐쇄망 사전점검) 통과 확인
  • STEP 4. pnpm install --frozen-lockfile 정상 확인
  • STEP 5. pnpm db:generate (Prisma 엔진 미러) 정상 확인
  • STEP 6. pnpm build (turbo) 성공 확인 — 이전 이슈 단계
  • STEP 7. Docker 이미지 build & push (api/web/worker/migrate) → ACR
  • STEP 8. Helm/ArgoCD 배포 (Pod 기동)

1. 이번에 반영된 변경 사항

metis.ai 빌드 성공본의 실제 설치 버전에 맞춰, 간소화버전 리포의 모든 package.json을 정확히 고정(^/~ 제거)했습니다. 이렇게 하면 Nexus가 항상 동일 버전을 내려주어 빌드 재현성이 보장됩니다.

※ 전체 버전 표는 본 리포의 각 package.json에서 확인 가능합니다(루트, apps/api, apps/web, apps/worker, packages/database, packages/types). packages/config는 의존성이 없어 변경 없음.

pnpm 10/11 빌드 스크립트 승인 (추가됨)

pnpm 10부터 보안상 패키지 postinstall(빌드 스크립트)이 기본 차단됩니다. tsx가 사용하는 esbuild는 플랫폼 바이너리 설치 스크립트가 필요하므로, 루트 package.json에 다음을 추가해 두었습니다.

// package.json (root)
"pnpm": {
  "onlyBuiltDependencies": [ "esbuild" ]
}

Prisma는 사내망에서 엔진 바이너리를 직접 반입/미러로 처리하시므로, prisma postinstall은 차단된 상태로 두고 pnpm db:generate를 명시 실행하는 현재 방식을 유지합니다.

2. 사내망 잔여 단계 — 상세 절차

1

Nexus 허용 목록(allow-list) 갱신 선행 필수

목적

버전을 고정·상향했으므로, 폐쇄망 Nexus 프록시가 새 버전들을 내려줄 수 있어야 합니다. 허용되지 않은 버전이 하나라도 있으면 install 단계에서 404/403으로 실패합니다.

확인할 항목

  • pnpm 자체: corepack이 pnpm@11.7.0 바이너리를 npm 레지스트리(Nexus)에서 받습니다 → pnpm@11.7.0 허용 필요
  • 상향된 라이브러리: turbo 2.9.18, @nestjs/cli 11.0.14, @nestjs/schematics 11.0.10, next 15.5.9, react/react-dom 19.2.4, typescript 5.9.3, prettier 3.8.3 등 — 고정 버전 전부
  • 전이 의존성: 위 패키지들의 새 하위 의존성(esbuild 플랫폼 패키지, @next/swc-* 등)도 Nexus 프록시 캐시 대상에 포함
팁 Nexus가 npm-proxy(원격 프록시) 형태라면 최초 요청 시 자동 캐시되지만, allow-list/방화벽 정책이 걸려 있으면 신규 버전이 막힐 수 있습니다. STEP 2 실행 전에 정책 담당과 확인하세요.
2

pnpm-lock.yaml 재생성 + 커밋 이슈 직접 원인

목적

package.json 버전을 바꿨으므로 기존 pnpm-lock.yaml은 더 이상 일치하지 않습니다. CI·Azure·Dockerfile 모두 pnpm install --frozen-lockfile을 쓰기 때문에, 락파일을 갱신하지 않으면 “lockfile not up to date / specifiers mismatch”로 빌드가 바로 실패합니다. (이전 보고의 “pnpm build 이슈” 핵심 원인)

실행 (Nexus 연결된 환경에서 1회)

corepack enable                         # package.json의 pnpm@11.7.0 사용
pnpm config set registry "https://nexus.corp.local/repository/npm-proxy/"
pnpm install                            # --frozen-lockfile 없이 → 락파일 재생성
git add pnpm-lock.yaml package.json apps/*/package.json packages/*/package.json
git commit -m "chore: bump pnpm 11.7.0 & pin deps (사내망 빌드 정렬)"
git push
주의 이 단계는 네트워크(Nexus)가 연결된 빌드 서버/러너에서 수행해야 합니다. 개발 PC가 인터넷 차단이라면 사내 빌드 러너에서 실행 후 락파일을 커밋하세요. 락파일은 lockfileVersion 9.0(pnpm 9~11 호환)이라 포맷 변경은 없습니다.

확인

pnpm-lock.yaml diff에 신규 버전(예: next@15.5.9, turbo@2.9.18)이 반영됐는지, lockfileVersion: '9.0' 유지되는지 확인.

3

폐쇄망 Preflight 점검

목적

CI 최초 잡(scripts/preflight-closed-network.sh)이 베이스 이미지·npm·prisma 미러 접근성을 사전 점검합니다. 로컬에서 미리 돌려 환경 변수가 올바른지 확인하세요.

bash scripts/preflight-closed-network.sh

필요 환경 변수 (CI Variables)

  • NEXUS_NPM_REGISTRY — 예: https://nexus.corp.local/repository/npm-proxy/
  • NEXUS_DOCKER_PROXY — 베이스 이미지(node:20-bookworm-slim) 프록시
  • PRISMA_MIRROR — Prisma 엔진 미러(raw-proxy)
4

의존성 설치 검증 (frozen-lockfile)

STEP 2에서 커밋한 락파일로 재현 설치가 되는지 확인합니다. CI와 동일한 명령입니다.

corepack enable
npm config set registry "$NEXUS_NPM_REGISTRY"
pnpm config set registry "$NEXUS_NPM_REGISTRY"
pnpm install --frozen-lockfile
esbuild 설치 로그에 esbuild 빌드가 실행되면 정상(루트 onlyBuiltDependencies 반영). “Ignored build scripts: esbuild” 경고가 보이고 이후 db:seed가 실패하면 STEP 트러블슈팅 참고.
5

Prisma Client 생성 (엔진 미러)

사내망은 Prisma 엔진 다운로드가 제약되므로, 미러/반입 바이너리를 사용합니다.

PRISMA_ENGINES_MIRROR="$PRISMA_MIRROR" pnpm db:generate

※ pnpm db:generate = prisma generate --schema=prisma/schema.prisma. 엔진 파일을 직접 반입하는 기존 방식이 있다면 그대로 사용하세요.

6

전체 빌드 (turbo)

이전에 막혔던 단계입니다. STEP 2 락파일 정렬 후에는 정상 통과해야 합니다.

pnpm build     # turbo run build → api(nest) · web(next) · worker/database/types(tsc)
만약 여기서 타입/컴파일 오류가 나면 버전 상향(특히 next 15.5.x, typescript 5.9.x, @nestjs/cli 11)에 따른 것일 수 있습니다. 오류 로그를 그대로 공유해 주시면 개별 대응하겠습니다. (현재 코드 기준 알려진 차단 이슈는 없습니다.)
7

Docker 이미지 빌드 & ACR 푸시

멀티타깃 Dockerfile(deploy/docker/Dockerfile)로 4개 타깃을 빌드합니다: api · web · worker · migrate. 베이스 이미지·레지스트리·Prisma 미러를 build-arg로 주입합니다.

# ACR 로그인
echo "$REGISTRY_PASSWORD" | docker login "$REGISTRY" -u "$REGISTRY_USERNAME" --password-stdin

# 4개 타깃 빌드 & 푸시
for T in api web worker migrate; do
  docker build -f deploy/docker/Dockerfile --target "$T" \
    --build-arg BASE_IMAGE="$NEXUS_DOCKER_PROXY/library/node:20-bookworm-slim" \
    --build-arg NPM_REGISTRY="$NEXUS_NPM_REGISTRY" \
    --build-arg PRISMA_ENGINES_MIRROR="$PRISMA_MIRROR" \
    -t "$REGISTRY/metis-$T:$TAG" .
  docker push "$REGISTRY/metis-$T:$TAG"
done
중요 이미지 안에서도 pnpm install --frozen-lockfile이 다시 실행됩니다. 따라서 STEP 2의 갱신된 pnpm-lock.yaml이 커밋·반영된 상태여야 Docker 빌드가 통과합니다.
8

배포 (Helm / ArgoCD)

CI는 Azure 로그인 후 Helm으로 배포하며, 이미지 레지스트리/태그를 주입합니다. ArgoCD를 쓰는 경우 매니페스트의 이미지 태그를 $TAG로 갱신 후 동기화합니다.

# Helm (CI 방식)
helm upgrade --install metis ./deploy/helm \
  --set image.registry="$REGISTRY" --set image.tag="$TAG"

# 또는 ArgoCD
argocd app set metis --helm-set image.tag="$TAG"
argocd app sync metis

배포 후 확인

  • migrate Job/Pod이 먼저 완료(pnpm db:push && pnpm db:seed)되는지
  • api / web / worker Pod Running + readiness 통과
  • 웹 접속 → 대시보드/Agent 실행 정상 동작

3. 트러블슈팅 (자주 나는 오류)

증상원인조치
ERR_PNPM_OUTDATED_LOCKFILE / specifiers don't match package.json 버전은 바뀌었는데 pnpm-lock.yaml 미갱신 STEP 2 수행(락파일 재생성·커밋). 임시로 --no-frozen-lockfile로 확인 가능하나, CI엔 갱신된 락파일 커밋이 정답
install 중 404 / 403 (특정 패키지·버전) Nexus allow-list에 신규 고정 버전 미등록 STEP 1 — 해당 버전을 Nexus 정책에 허용
Cannot find module 'pnpm' / corepack 다운로드 실패 corepack이 Nexus에서 pnpm@11.7.0을 못 받음 Nexus npm 프록시에 pnpm@11.7.0 허용. 또는 베이스 이미지에 pnpm 사전 설치
esbuild: Failed to install platform binary / seed 단계 esbuild 오류 pnpm 10+ 빌드 스크립트 차단 루트 pnpm.onlyBuiltDependencies: ["esbuild"] 반영 확인(이미 추가됨). 그래도 막히면 pnpm approve-builds
Prisma Failed to fetch the engine 엔진 다운로드가 폐쇄망에서 차단 PRISMA_ENGINES_MIRROR 설정 또는 엔진 바이너리 직접 반입
turbo / next build 타입 오류 버전 상향(next 15.5.x, ts 5.9.x, nest cli 11)에 따른 차이 오류 로그 공유 → 개별 대응 (현재 코드 기준 알려진 차단 이슈 없음)

핵심 요약 한 줄

지금 막 할 일은 단 하나 — “Nexus 연결된 러너에서 pnpm install로 pnpm-lock.yaml을 재생성해 커밋”입니다. 그러면 --frozen-lockfile 기반의 CI·Docker 빌드가 정상 진행됩니다. (그 전에 STEP 1의 Nexus 버전 허용만 확인)

문의: 본 변경(버전 고정·pnpm 11.7.0·onlyBuiltDependencies)은 모두 리포에 반영 완료. 빌드 로그에서 막히는 지점이 있으면 해당 로그를 전달 주시면 이어서 대응하겠습니다.
