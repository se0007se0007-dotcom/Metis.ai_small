METIS.AI 사용자 매뉴얼

멀티테넌트 AgentOps 거버넌스 SaaS — Agent 실행·평가·정책·릴리스를 한 곳에서 관리합니다.

1. 제품 개요

METIS.AI는 기업이 다수의 AI Agent를 안전하고 일관되게 운영하도록 돕는 거버넌스 플랫폼입니다. Agent의 실행 결과를 4-Gate(품질·보안·이상탐지·비용)로 평가하고, 정책으로 통제하며, 릴리스(Canary/Shadow/Replay)를 관리합니다.

핵심 가치

  • 평가 자동화: 모든 Agent 응답을 4-Gate로 채점
  • 정책 기반 통제: 코드 수정 없이 설정으로 Gate 기준 변경
  • 피드백 루프: 이력 분석 → 정책 자동 제안 → 승인 반영
  • 안전한 릴리스: Canary/Shadow/Replay로 점진 배포

주요 사용자

  • 운영자(OPERATOR): Agent 실행·모니터링
  • 개발자(DEVELOPER): 워크플로우·Agent 등록
  • 감사자(AUDITOR): 로그·정책·평가 검토
  • 테넌트 관리자(TENANT_ADMIN): 정책·사용자 관리
  • 플랫폼 관리자(PLATFORM_ADMIN): 전체 운영

2. 설치 및 실행

Windows 환경 기준, 사전 요구사항: Node.js 20+, pnpm, Docker Desktop.

원클릭 실행

  1. 프로젝트 루트의 start-metis.bat 더블클릭
  2. 배치가 자동으로: Docker(PostgreSQL+Redis) 기동 → 의존성 설치 → Prisma 클라이언트 생성 → DB 스키마 push → seed → 빌드 → API/Worker/Frontend 3개 창 기동
  3. 브라우저가 http://localhost:3000 자동 오픈
서비스 주소 설명
Frontend http://localhost:3000 웹 UI
API http://localhost:4000/v1 REST API (prefix /v1)
Swagger 문서 http://localhost:4000/docs OpenAPI UI (개발 모드만)
새 기능 추가 후에는 새 DB 테이블이 생길 수 있으므로 start-metis.bat를 재실행하거나 pnpm db:generate && pnpm --filter @metis/database push -- --accept-data-loss를 먼저 실행하세요.
중지: 실행된 콘솔 창에서 아무 키나 누르면 3개 서버가 모두 종료됩니다(stop-metis.bat도 사용 가능).

3. 로그인 · 권한

기본 관리자 계정: admin@metis.ai / metis1234. 로그인 시 JWT 액세스 토큰이 발급되어 HttpOnly 쿠키와 응답 body로 전달됩니다.

4. 화면 구성

상단 네비게이션(TopNav) + 좌측 사이드바(SideNav) + 중앙 메인 + (일부 화면) 우측 인스펙터 구조입니다. 좌측 사이드바는 기능 그룹별로 묶여 있습니다: 대시보드 · Agent · 미션 · Governance · Knowledge · Orchestration · Insights · Release · Admin.

5. Agent 카탈로그

좌측 Agent 그룹에서 카테고리별 Agent를 확인합니다. 기본 제공 14개 Agent:

그룹 Agent
운영(OPS) 7종 테스트 자동화, 서비스 모니터링, 캠페인 모니터링, 변경 영향도, 이벤트 대응, 지식 자산화, 품질가디언
개발(DEV) 4종 Spec Agent, 영향도 분석, Dev Agent, Test Agent
편의(EXT) 3종 QueryBuddy, SR Routing, SR 영향도

6. Agent 실행 테스트

좌측 Agent 실행 테스트에서 특정 Agent를 시뮬레이션 실행하고 그 결과가 자동으로 4-Gate 평가됩니다. 실행 세션과 스텝이 DB에 기록되어 모니터링·이력에서 추적됩니다.

7. 미션 (A2A 협업)

여러 Agent가 하나의 목표를 위해 메시지를 주고받으며 협업하는 작업 단위입니다. 활성 미션 / 미션 이력에서 진행 상태, 참여 Agent, 핸드오프, 자가치유(Auto-Action)를 확인합니다.

  • 미션 상태: PLANNING → RUNNING → (WAITING_HUMAN) → SUCCEEDED/FAILED/CANCELLED/ROLLED_BACK
  • 미션 상세 화면에서 참여 Agent와 타임라인을 봅니다.

8. AI 활동 로그 (Governance)

Governance → AI 활동 로그: 모든 감사 이벤트(생성·실행·정책검사·인증·릴리스 등)를 조회합니다. action·correlationId로 필터링하고 페이지네이션됩니다.

9. 정책코드 엔진

Governance → 정책코드 엔진: 실행 차단/경고 규칙(YAML/JSON)을 정의합니다. 규칙은 conditions + effect(deny/warn/allow) 구조이며, 실행 전 정책 평가에서 FAIL이면 차단됩니다.

10. 평가 Gate 설정 Phase 1

Governance → 평가 Gate 설정: 4-Gate 평가 엔진의 가중치·임계값을 코드 수정 없이 조정합니다. 저장하면 5분 내(캐시) 평가 엔진에 반영됩니다.

조정 항목

Gate 주요 설정
품질 가중치, Hard Gate 최소 점수, LLM Judge 사용/모델/일일 예산
보안 가중치, Critical/High 상한(cap), PII 스캔, 프롬프트 인젝션 탐지
이상탐지 가중치, Z-Score 임계값, IQR 계수
비용 가중치, 일일 예산, Slow/Critical 지연(ms)
연동 Canary 품질/보안 최소, ORB 통과/조건부 임계값
  1. 상단 드롭다운에서 적용 범위 선택(기본 / 운영 / 개발 / 고도화)
  2. 각 Gate 카드에서 값 조정 — 가중치 합계는 1.00이어야 저장 가능
  3. 저장 클릭 → 토스트 확인. 되돌리려면 기본값 복원
정책이 없거나 기본값이면 평가 점수는 기존 동작과 100% 동일합니다(하위호환).

11. 정책 제안 (피드백 루프) Phase 2

Governance → 정책 제안: 최근 평가 이력을 분석해 정책 조정을 제안하고, 관리자가 승인하면 정책에 자동 반영합니다.

탐지 패턴

패턴 의미 제안
보안 반복 실패 특정 Agent가 보안 Gate 반복 실패 securityCriticalCap 강화
품질 하락 품질 점수 하락 추세 qualityHardGateMin 상향
비용 초과 비용 효율 지속 저조 검토 권고(자동 변경 없음)
이상 급증 이상 탐지율 임계 초과 LLM Judge 활성화 권고
  1. 지금 분석 클릭 → 최근 30일 이력 점검
  2. 제안 카드에서 변경안 diff 확인 (예: securityCriticalCap: 40 → 30)
  3. 승인·적용 또는 거부
  4. 상단 적응형 샘플링 현황: Agent별 LLM Judge 호출 빈도(%)를 표시 — 이상 시 100%로 올리고 정상 지속 시 10%까지 자동 절감
자동으로 정책을 바꾸지 않습니다. 반드시 제안 → 관리자 승인을 거칩니다. 비용 초과처럼 위험 낮은 패턴은 자동 변경 없이 권고만 합니다.

12. ORB 심사 (Ops.AI Review Board)

Governance → ORB 심사: Agent 등록 전 5개 영역(품질30·성능20·보안25·데이터표준15·확장성10 = 100점)으로 심사합니다. 필수 체크(M1~M7) 중 하나라도 실패하면 점수와 무관하게 자동 반려됩니다.

  1. 심사 목록에서 대상 클릭 → 5개 영역 항목별 1~5점 채점
  2. 필수 체크(Mandatory) 통과 여부 토글
  3. 판정: 70+ 승인 / 50–69 조건부 / 그외 반려
  4. 강점·개선점 입력 후 최종 판정 제출

18. Agent Evaluator 핵심

Insights → Agent Evaluator: 4-Gate 평가 결과를 종합 대시보드로 봅니다. 탭: 개요 · 품질 · 보안 · 이상 · 비용.

4-Gate 평가 모델

Gate 측정 기본 가중치
품질 정확도·환각률·응답품질 (LLM Judge 연계) 40%
보안 입력 위협·출력 유출·툴체인 위험 30%
비용 비용 효율·지연·토큰 15%
이상탐지 Z-Score·IQR·추세·오류급증 15%
드릴다운: "품질" 탭의 최근 평가 상세 표에서 행을 클릭하면 해당 건의 4-Gate 상세(점수·위험도·비용·이상·권고)가 펼쳐집니다.

수동 평가 데모

"개요" 탭의 데모 패널에서 입력/출력/컨텍스트를 넣고 즉시 평가해볼 수 있습니다(POST /evaluator/demo). LLM Judge는 .env에 ANTHROPIC_API_KEY 또는 OPENAI_API_KEY가 있을 때 동작합니다.

LLM Judge

응답을 사실정확도·완전성·관련성·독성·편향(+컨텍스트 시 충실도) 5~6개 차원으로 채점합니다. 사실 오류는 factual_accuracy=0으로 품질 점수가 강하게 제한되고, "완료" 같은 한 단어 응답은 completeness=0으로 처리됩니다.

24. 테스트 가이드

레포 scripts/에 통합 스모크 스크립트가 있습니다(서버 기동 후 별도 터미널에서 실행). 

cd metis-ai
# 평가 Gate 정책 API
node scripts/test-evaluation-policy.mjs
# 정책 피드백(제안/승인/샘플링)
node scripts/test-policy-feedback.mjs --approve
# E2E 전수 평가 (14×4×3=168건, LLM Judge는 .env 키 필요)
node scripts/test-e2e-evaluation.mjs --repeat 3 --out e2e-report.json
각 스크립트는 로그인 → API 호출 → 결과를 ✅/❌로 출력합니다. 계정/포트가 다르면 --email --password --base 플래그로 지정하세요.

25. FAQ · 문제해결

Q. "평가 Gate 설정" 저장 시 오류가 났습니다.

최신 버전에서 수정되었습니다. 저장 페이로드에서 읽기전용 id를 제외하도록 고쳐졌습니다. 프론트를 새로고침하세요.

Q. "정책 제안"에서 "table does not exist" 에러.

새 테이블(PolicySuggestion)이 추가됐기 때문입니다. start-metis.bat 재실행 또는 pnpm db:generate && pnpm --filter @metis/database push -- --accept-data-loss 후 다시 시도하세요.

Q. LLM Judge가 동작하지 않습니다(L0만 표시).

.env에 ANTHROPIC_API_KEY 또는 OPENAI_API_KEY를 설정해야 합니다. 없으면 통계 기반(Layer 0)만 적용되어 환각 판정 정확도가 낮아집니다.

Q. 제안이 0건으로 나옵니다.

평가 이력이 적거나 패턴이 없으면 정상입니다. Agent 실행 테스트나 POST /evaluator/demo로 평가를 여러 건 쌓은 뒤 "지금 분석"을 누르세요.

Q. Swagger가 안 보입니다.
